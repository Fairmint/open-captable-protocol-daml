# Task: CouponMinter Contract Design

**Date**: 2026-01-12
**Status**: Open
**Target Repo**: `open-captable-protocol-daml`
**Related ADR**: [ADR-003: Value-Based Coupon Minting for OCP Transactions](../../../docs/adr/003-featured-app-markers-for-ocp-transactions.md)

## Goal

Design and implement a simple `CouponMinter` contract that allows the backend service to mint Featured App Activity Markers on demand. The contract is intentionally minimal—all throttling, timing, and business logic resides in the backend CouponMinter service.

## Background

Per ADR-003, we need rate-limited coupon minting for OCP transactions. Rather than embedding complex rate-limiting logic in DAML contracts, we keep the contract simple and push all scheduling logic to the backend service. This provides:

- **Simplicity**: The contract is easy to audit and understand
- **Flexibility**: Throttling parameters can be adjusted without contract upgrades
- **Observability**: The backend service can log and monitor minting activity

## Proposed Design

### Contract: `CouponMinter`

A stateless utility contract that mints a specified number of activity markers in a single transaction.

```daml
module Fairmint.OpenCapTable.CouponMinter where

-- CouponMinter - Utility contract for minting Featured App Activity Markers
-- Allows the backend CouponMinter service to mint markers on demand
-- All throttling and scheduling logic resides in the backend service

import DA.List (replicateA)
import Splice.Api.FeaturedAppRightV1 (FeaturedAppRight(..), FeaturedAppRight_CreateActivityMarker(..), AppRewardBeneficiary(..), FeaturedAppActivityMarker)

-- | Result of minting coupons
data MintCouponsResult = MintCouponsResult with
    markerCids: [ContractId FeaturedAppActivityMarker]
      -- ^ Contract IDs of all created activity markers
  deriving (Eq, Show)

-- | CouponMinter contract
-- Stateless utility for minting Featured App Activity Markers
template CouponMinter
  with
    operator: Party
      -- ^ System operator (Fairmint)
    provider: Party
      -- ^ Provider party for the featured app (typically same as operator)
  where
    signatory operator
    observer provider

    -- | Mint activity markers for Featured App rewards
    -- All throttling and rate-limiting is handled by the backend service
    nonconsuming choice MintCoupons : MintCouponsResult
      with
        actor: Party
          -- ^ Party executing this choice (must be operator or provider)
        featuredAppRight: ContractId FeaturedAppRight
          -- ^ FeaturedAppRight contract to use for minting
        count: Int
          -- ^ Number of markers to create (must be >= 1)
        metadata: Text
          -- ^ Description/justification for this minting operation
          -- Examples: "StockIssuance:abc123:$5000", "Batch:xyz789:50coupons"
        beneficiaries: [AppRewardBeneficiary]
          -- ^ Reward beneficiaries (typically the app provider)
      controller actor
      do
        -- Validate inputs
        assertMsg "count must be at least 1" (count >= 1)
        assertMsg "actor must be operator or provider" (actor == operator || actor == provider)
        
        -- Mint the requested number of markers
        allMarkerCids <- replicateA count $ do
          result <- exercise featuredAppRight FeaturedAppRight_CreateActivityMarker with
            beneficiaries
          pure result.activityMarkerCids
        
        pure MintCouponsResult with
          markerCids = concat allMarkerCids
```

### Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| **Stateless contract** | No state to manage; contract is a capability/authorization |
| **`nonconsuming` choice** | Contract persists across multiple minting operations |
| **`actor` parameter** | Supports both operator and provider as controllers |
| **`metadata` field** | Audit trail linking markers to source transactions |
| **`beneficiaries` parameter** | Passed through to Splice API; typically the app provider |
| **No throttling in contract** | Backend service handles rate limiting per ADR-003 |
| **Batch minting in single tx** | Efficient for small batches; backend controls batch size |

### Metadata Format Convention

The `metadata` field provides an audit trail. Recommended format:

```
<TransactionType>:<TransactionId>:<Value>
```

Examples:
- `"StockIssuance:txn-abc123:$5000"` — 50 markers for a $5,000 stock issuance
- `"ConvertibleIssuance:txn-xyz789:$100000"` — 1,000 markers for a SAFE
- `"BatchUpdate:batch-456:$12500"` — 125 markers for a batch operation
- `"Manual:admin-request:Promotional grant"` — Manual minting with justification

### Contract Lifecycle

```
┌─────────────────────────────────────────────────────────────┐
│                    BACKEND SERVICE                          │
│                                                             │
│  1. Calculate coupon count from transaction value           │
│  2. Apply rate limiting (TPS_LIMIT from ADR-003)            │
│  3. Schedule minting operations                             │
│  4. Call MintCoupons choice with appropriate batch size     │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    COUPON MINTER CONTRACT                   │
│                                                             │
│  - Validates inputs (count >= 1, actor authorized)          │
│  - Exercises FeaturedAppRight_CreateActivityMarker          │
│  - Returns created marker contract IDs                      │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    SPLICE FEATURED APP API                  │
│                                                             │
│  - Creates FeaturedAppActivityMarker contracts              │
│  - Markers earn rewards in Canton Network rounds            │
└─────────────────────────────────────────────────────────────┘
```

## Alternative: Factory Pattern

If we need to create separate `CouponMinter` contracts per issuer (for attribution or access control), we could add a factory:

```daml
template CouponMinterFactory
  with
    operator: Party
  where
    signatory operator

    choice CreateCouponMinter : ContractId CouponMinter
      with
        provider: Party
      controller operator
      do
        create CouponMinter with ..
```

**Recommendation**: Start with a single shared `CouponMinter` contract. Add factory pattern only if per-issuer isolation is needed.

1. **Per-issuer tracking**: If we need to track coupons per issuer, consider adding `issuerId: Text` to metadata format
2. **Audit contract**: If full on-chain audit trail is needed, consider creating a separate `CouponMintingRecord` contract
