# Task: Contract Generation Templates

**Date**: 2026-01-12
**Status**: Open
**Target Repo**: `open-captable-protocol-daml`

## Goal

Replace programmatic string building in `generate-captable.ts` with a template-based approach using placeholder files. This makes the generated contract structure visible and reviewable directly in GitHub.

## Background

The current `generate-captable.ts` builds DAML code by concatenating strings in TypeScript:

```typescript
function generateProcessCreate(types: TypeDef[]): string {
  const cases = sortedTypes.map((t) => generateCreateCase(t)).join("\n");
  return `processCreate ctx maps createData = case createData of
${cases}`;
}
```

This approach has several drawbacks:
- **Hard to review**: The actual contract structure is buried in TypeScript string interpolation
- **Hard to understand at a glance**: Must read/run the generator to see what choices exist
- **Error-prone**: String building makes typos and formatting errors easy to miss
- **IDE support lost**: No DAML syntax highlighting or validation in template strings

The project previously used Mustache templates but moved away due to complexity. Now that the contracts are nearly complete and stable, it's time to revisit a template-based approach.

## Current State

### What We Generate

`CapTable.daml` contains:
- 1 template (`CapTable`)
- 48+ OCF type maps (stakeholders, stock_classes, etc.)
- 1 batch choice (`UpdateCapTable`)
- Sum types (`OcfCreateData`, `OcfEditData`, `OcfObjectId`)
- Helper functions (`processCreate`, `processEdit`, `processDelete`, etc.)

### Current Generator Flow

```
captable-config.yaml → generate-captable.ts → CapTable.daml
                              ↓
                    OCF/*.daml files (type discovery)
```

## Proposed Solution

### Template Structure

Create template files with clear placeholders that show exactly what the generated output will look like:

```
scripts/codegen/templates/
├── CapTable.daml.template        # Main template with all sections
├── partials/
│   ├── imports.daml.partial      # OCF type imports
│   ├── sum-types.daml.partial    # OcfCreateData, OcfEditData, OcfObjectId
│   ├── helper-types.daml.partial # CapTableMaps, toMaps, etc.
│   ├── helper-functions.daml.partial # processCreate, processEdit, processDelete
│   └── batch-choice.daml.partial # UpdateCapTable choice
└── loops/
    ├── create-case.daml.loop     # Single case in processCreate
    ├── edit-case.daml.loop       # Single case in processEdit
    └── delete-case.daml.loop     # Single case in processDelete
```

### Main Template Example

`CapTable.daml.template`:

```daml
module Fairmint.OpenCapTable.CapTable where

-- =============================================================================
-- CapTable Contract (GENERATED - DO NOT EDIT)
-- =============================================================================
-- Stateful cap table with batch UpdateCapTable choice for efficient bulk updates.
-- See ADR-002: Stateful Cap Table with OCF Object References
--
-- Generated by: scripts/codegen/generate-captable.ts
-- Template: scripts/codegen/templates/CapTable.daml.template
-- Config: scripts/codegen/captable-config.yaml
-- =============================================================================

import DA.Map (Map)

import Fairmint.OpenCapTable.Types (Context)
import Fairmint.OpenCapTable.Helpers (createMarker)
import Fairmint.OpenCapTable.OCF.Issuer (Issuer(..), IssuerOcfData)

-- OCF Types
{{#each types}}
import {{module}} ({{name}}(..), {{data_type}})
{{/each}}


-- =============================================================================
-- Batch Operation Types
-- =============================================================================

{{> sum-types}}


-- =============================================================================
-- Internal Helper Types and Functions
-- =============================================================================

{{> helper-types}}

{{> helper-functions}}


-- =============================================================================
-- CapTable Template
-- =============================================================================

template CapTable
  with
    context: Context

    -- Issuer (exactly 1, edit only - no create/delete)

{{> batch-choice}}
```

### Template Engine Options

Several options to evaluate:

1. **Handlebars** (recommended)
   - Mature, well-supported
   - Partials and helpers built-in
   - Works well with TypeScript
   - `npm install handlebars`

2. **EJS**
   - Allows JavaScript in templates
   - More flexible but less structured
   - `npm install ejs`

3. **Nunjucks**
   - Python Jinja2 inspired
   - Good template inheritance
   - `npm install nunjucks`

4. **Custom minimal engine**
   - Simple find/replace for placeholders
   - No external dependency
   - Limited flexibility

### Recommended Approach: Handlebars

Handlebars provides a good balance of:
- Readable templates (close to final output)
- Powerful enough for loops and conditionals
- Partials for code reuse
- Strong TypeScript support

  const context = {
    types: types.sort((a, b) => a.tier - b.tier || a.name.localeCompare(b.name)),
    types_alphabetical: [...types].sort((a, b) => a.name.localeCompare(b.name)),

## Benefits

1. **Reviewable in GitHub**: Templates show actual DAML structure with placeholders
2. **Easier to modify**: Change templates, not TypeScript string logic
3. **Better separation**: Config → Types → Templates → Output
4. **IDE support**: DAML syntax highlighting in template files (with minor noise from placeholders)
5. **Onboarding**: New contributors can understand the contract at a glance

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Template engine complexity | Start with Handlebars, well-documented and widely used |
| Whitespace/formatting issues | Strict testing against current output |
| Over-engineering | Keep templates simple, avoid too many partials |
| Build-time dependency | Handlebars is stable, minimal footprint |
