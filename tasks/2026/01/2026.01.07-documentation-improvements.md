# Task: Documentation Improvements

**Date**: 2026-01-07
**Status**: Open
**Target Repo**: `open-captable-protocol-daml`

## Goal

Improve documentation completeness, consistency, and accessibility to enable faster onboarding and reduce friction for contributors and SDK users.

## Background

The repository has good foundational documentation (`llms.txt`, `README.md`, task files), but several areas need improvement:
- Incomplete design documents
- Missing API documentation
- ADRs not consistently updated
- No usage examples for SDK consumers

## Current State

### What's Good
- ✅ Excellent `llms.txt` as AI/developer entry point
- ✅ Clear coding standards documented
- ✅ Task file format well-established
- ✅ ADR-002 provides architectural context

### Gaps Identified

1. **PersonalAirdrop Documentation Incomplete**
   - `PERSONAL_AIRDROP_DESIGN.md` notes "TypeScript SDK: Not yet generated"
   - No usage scripts exist despite detailed design

2. **Missing ADRs**
   - No ADR for the code generation approach (`generate-captable.ts`)
   - No ADR for PaymentStream architecture
   - ADR-001 referenced but lives in different repo

3. **No API Reference**
   - Generated TypeScript types have no documentation
   - Contract choices lack usage examples
   - No SDK quick-start guide

4. **Inconsistent README Structure**
   - Root `README.md` differs in format from package READMEs
   - Some packages have sparse or missing READMEs

5. **No Contribution Guide**
   - No `CONTRIBUTING.md`
   - No PR template
   - Issue templates exist but could be improved

## Documentation Standards

### ADR Format
```markdown
# ADR-XXX: Title

## Status
Proposed | Accepted | Deprecated | Superseded

## Context
What is the issue that we're seeing that motivates this decision?

## Decision
What is the change we're proposing?

## Consequences
What becomes easier or harder to do?
```

### README Sections
1. Overview (1-2 sentences)
2. Contracts/Features
3. Installation/Usage
4. Development
