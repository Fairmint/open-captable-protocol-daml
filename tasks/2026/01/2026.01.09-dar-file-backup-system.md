# Task: DAR File Backup System

**Date**: 2026-01-09 **Status**: ✅ Completed **Target Repo**: `open-captable-protocol-daml`

## Goal

Implement a system to preserve and version DAR files that are uploaded to mainnet. Deterministic
builds are only guaranteed when using the exact same compiler version, so we need to keep the exact
bytes of published packages to avoid rebuilding.

## Background

When uploading DAR files to mainnet, the exact bytes matter. DAML builds are only deterministic when
using the same compiler version. If we rebuild a package with a different SDK version, we get
different bytes even with identical source code. This breaks:

1. **Package verification** - Canton validates package hashes
2. **Reproducibility** - Cannot recreate the exact deployed artifact
3. **Rollback safety** - Cannot redeploy an old known-good version

The solution is to store published DAR files in the repository (like
[splice](https://github.com/hyperledger-labs/splice/tree/main/daml/dars)) with CI checks to prevent
accidental modifications.

### What's Missing

1. **No DAR preservation** - DARs are rebuilt on each build, losing published bytes
2. **No integrity checks** - CI doesn't detect DAR changes
3. **No versioned history** - Cannot track what was deployed when

## Proposed Solution

### 1. Create `dars/` Directory Structure

```
dars/
├── README.md                    # Documentation
├── dars.lock                    # Hash manifest for CI verification
├── OpenCapTable-v26/
│   └── 0.0.1/
│       ├── OpenCapTable-v26.dar
│       └── metadata.json        # Upload date, SDK version, networks
├── OpenCapTableReports-v01/
│   └── 0.0.1/
│       └── ...
└── CantonPayments/
    └── 0.0.1/
        └── ...
```

### 2. Create `dars.lock` Manifest

Similar to
[splice's dars.lock](https://github.com/hyperledger-labs/splice/blob/main/daml/dars.lock):

```json
{
  "version": 1,
  "packages": {
    "OpenCapTable-v26/0.0.1/OpenCapTable-v26.dar": {
      "sha256": "abc123...",
      "size": 12345,
      "sdkVersion": "3.3.0-snapshot.20250603.0",
      "uploadedAt": "2026-01-09T12:00:00Z",
      "networks": ["mainnet", "devnet"]
    }
  }
}
```

### 3. Configure Git LFS for DAR Files

Add to `.gitattributes`:

```
*.dar filter=lfs diff=lfs merge=lfs -text
```

**Benefits:**

- Git repo stays fast (DAR files are 5-15MB each)
- Full history preserved without bloating repo
- GitHub displays LFS files cleanly

### 4. Add CI Check for DAR Integrity

New workflow `.github/workflows/check-dars.yml`:

```yaml
name: Check DAR Integrity

on:
  pull_request:
    paths:
      - 'dars/**'
      - '.github/workflows/check-dars.yml'

jobs:
  check-dars:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          lfs: true

      - name: Verify DAR hashes
        run: |
          tsx scripts/verify-dars.ts

      - name: Check for unauthorized DAR changes
        run: |
          # Fail if any DAR file changed without corresponding lock update
          tsx scripts/check-dar-changes.ts
```

### 5. Create Scripts

#### `scripts/backup-dar.ts`

Backs up a DAR after successful upload:

```typescript
#!/usr/bin/env node
// Backup a DAR file to dars/ after mainnet upload
// Usage: tsx scripts/backup-dar.ts --package OpenCapTable-v26 --version 0.0.1
```

**Behavior:**

- Copies DAR from `.daml/dist/` to `dars/{package}/{version}/`
- Computes SHA256 hash
- Updates `dars.lock` with metadata
- Fails if DAR already exists (no overwrites)

#### `scripts/verify-dars.ts`

CI script to verify DAR integrity:

```typescript
#!/usr/bin/env node
// Verify all DAR files match their recorded hashes
```

#### `scripts/check-dar-changes.ts`

CI script to prevent unauthorized DAR modifications:

```typescript
#!/usr/bin/env node
// Fail PR if DAR changed without explicit version bump
```

### 6. Update Upload Scripts

Modify upload scripts to:

1. Check if DAR already exists in `dars/` - if so, use it
2. After successful mainnet upload, call `backup-dar.ts`
3. Emit warning if building fresh instead of using backed-up DAR

## Edge Cases to Handle

1. **Fresh deploy with no backup** - Build fresh, backup after upload
2. **SDK version mismatch** - Warn if current SDK differs from backup metadata
3. **Multiple networks** - Track which networks have received the DAR
4. **Concurrent uploads** - Lock file should handle merge conflicts gracefully

## Security Considerations

1. **Hash verification** - Always verify hashes before upload
2. **No manual DAR modifications** - CI prevents direct DAR changes
3. **Audit trail** - Git history + metadata.json provide full lineage
