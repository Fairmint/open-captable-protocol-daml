# Task: Security ID Validation for Tier 4 Transactions

**Date**: 2026-01-21 **Status**: Open **Target Repo**: `open-captable-protocol-daml`

## Goal

Add validation that `security_id` references exist before creating tier 4 transactions (transfers,
cancellations, exercises, etc.).

## Problem

Currently, the `captable-config.yaml` only validates specific references like `stakeholder_id` and
`stock_class_id` for issuance transactions. However, tier 4 transactions reference securities via
`security_id` without any validation.

**Example**: A `StockTransfer` with a non-existent `security_id` would be created without error:

```yaml
# Current config has NO validations for StockTransfer:
validations:
  StockIssuance: [stakeholder_id, stock_class_id]
  # StockTransfer: [] -- missing security_id validation
```

This means:

- `StockTransfer` can reference a non-existent `security_id`
- `StockCancellation` can cancel a security that doesn't exist
- `WarrantExercise` can exercise a warrant that was never issued

## Solution: Type-Specific Security Validation

Add validation rules in `captable-config.yaml` that check the appropriate issuance map based on
transaction type. This is consistent with how we validate other references (stakeholder_id,
stock_class_id) and ensures the security exists in the correct type-specific map.

### Config Changes

```yaml
validations:
  # Existing issuance validations
  StockIssuance: [stakeholder_id, stock_class_id]
  EquityCompensationIssuance: [stakeholder_id]
  ConvertibleIssuance: [stakeholder_id]
  WarrantIssuance: [stakeholder_id]

  # Stock transactions validate against stock_issuances.security_id
  StockTransfer: [security_id:stock_issuances]
  StockCancellation: [security_id:stock_issuances]
  StockAcceptance: [security_id:stock_issuances]
  StockRetraction: [security_id:stock_issuances]
  StockConversion: [security_id:stock_issuances]
  StockRepurchase: [security_id:stock_issuances]
  StockReissuance: [security_id:stock_issuances]
  StockConsolidation: [security_id:stock_issuances]

  # Warrant transactions validate against warrant_issuances.security_id
  WarrantTransfer: [security_id:warrant_issuances]
  WarrantCancellation: [security_id:warrant_issuances]
  WarrantAcceptance: [security_id:warrant_issuances]
  WarrantRetraction: [security_id:warrant_issuances]
  WarrantExercise: [security_id:warrant_issuances]

  # Convertible transactions validate against convertible_issuances.security_id
  ConvertibleTransfer: [security_id:convertible_issuances]
  ConvertibleCancellation: [security_id:convertible_issuances]
  ConvertibleAcceptance: [security_id:convertible_issuances]
  ConvertibleRetraction: [security_id:convertible_issuances]
  ConvertibleConversion: [security_id:convertible_issuances]

  # Equity compensation transactions validate against equity_compensation_issuances.security_id
  EquityCompensationTransfer: [security_id:equity_compensation_issuances]
  EquityCompensationCancellation: [security_id:equity_compensation_issuances]
  EquityCompensationAcceptance: [security_id:equity_compensation_issuances]
  EquityCompensationRetraction: [security_id:equity_compensation_issuances]
  EquityCompensationExercise: [security_id:equity_compensation_issuances]
  EquityCompensationRelease: [security_id:equity_compensation_issuances]
  EquityCompensationRepricing: [security_id:equity_compensation_issuances]

  # Vesting transactions - need to determine which issuance type to validate against
  # These may need special handling since vesting can apply to multiple security types
  VestingStart: [security_id:equity_compensation_issuances]
  VestingEvent: [security_id:equity_compensation_issuances]
  VestingAcceleration: [security_id:equity_compensation_issuances]
```

### Why This Approach

1. **Consistent with existing patterns**: We already validate `stakeholder_id` against `stakeholders`
   map and `stock_class_id` against `stock_classes` map. This follows the same pattern.

2. **Type-safe**: Validates that the security exists in the correct issuance type. A
   `WarrantTransfer` can only reference a warrant security, not a stock security.

3. **Simple implementation**: Only requires config changes and minor codegen template updates. No new
   data structures or maps needed.

4. **O(1) lookup**: Uses existing maps which are already O(1) for lookups.

### Codegen Changes Required

The current validation in `create-case.daml` looks up by `id` field:

```daml
assertMsg ("Stakeholder not found: " <> d.stakeholder_id) (isSome (Map.lookup d.stakeholder_id maps.stakeholders))
```

For `security_id` validation, we need to look up by `security_id` field in the issuance data, not by
`id`. This requires either:

**Option A**: Build a secondary index map (`security_id → ContractId`) at CapTable creation/update
time for each issuance type.

**Option B**: Extend the validation syntax to support field mapping:

```yaml
# field_to_check:map_name:lookup_field
StockTransfer: [security_id:stock_issuances:security_id]
```

Option A is cleaner but requires maintaining additional maps. Option B is more flexible but adds
complexity to the config syntax.

**Recommendation**: Use Option A - add `stock_issuances_by_security_id`, etc. maps that are
maintained alongside the primary maps. This keeps lookups O(1) and validation simple.
