# Task: Security ID Validation for Tier 4 Transactions

**Date**: 2026-01-21 **Status**: Open **Target Repo**: `open-captable-protocol-daml`

## Goal

Add validation that `security_id` references exist before creating tier 4 transactions (transfers,
cancellations, exercises, etc.).

## Problem

Currently, the `captable-config.yaml` only validates specific references like `stakeholder_id` and
`stock_class_id` for issuance transactions. However, tier 4 transactions reference securities via
`security_id` without any validation.

**Example**: A `StockTransfer` with a non-existent `security_id` would be created without error:

```yaml
# Current config has NO validations for StockTransfer:
validations:
  StockIssuance: [stakeholder_id, stock_class_id]
  # StockTransfer: [] -- missing security_id validation
```

This means:

- `StockTransfer` can reference a non-existent `security_id`
- `StockCancellation` can cancel a security that doesn't exist
- `WarrantExercise` can exercise a warrant that was never issued

## Proposed Solution

### Option A: Unified Securities Map (Recommended)

Add a unified `securities` map that tracks all active security IDs across all issuance types:

```daml
template CapTable
  with
    -- ... existing maps ...

    -- Unified security tracking (security_id → issuance type + ContractId)
    securities: Map Text SecurityRef
```

Where `SecurityRef` tracks which issuance type owns the security:

```daml
data SecurityRef
  = StockSecurityRef (ContractId StockIssuance)
  | EquityCompensationSecurityRef (ContractId EquityCompensationIssuance)
  | ConvertibleSecurityRef (ContractId ConvertibleIssuance)
  | WarrantSecurityRef (ContractId WarrantIssuance)
  deriving (Eq, Show)
```

**Pros:**

- O(1) lookup for any security_id
- Type-safe: know which issuance type the security belongs to
- Can validate that security type matches transaction type (e.g., StockTransfer only for stock
  securities)

**Cons:**

- Adds maintenance overhead: must update `securities` map on every issuance create/edit/delete
- More complex codegen template changes

### Option B: Type-Specific Security Validation

Add validation rules in `captable-config.yaml` that check the appropriate issuance map based on
transaction type:

```yaml
validations:
  # Stock transactions validate against stock_issuances
  StockTransfer: [security_id:stock_issuances]
  StockCancellation: [security_id:stock_issuances]
  StockAcceptance: [security_id:stock_issuances]
  # etc.

  # Warrant transactions validate against warrant_issuances
  WarrantTransfer: [security_id:warrant_issuances]
  WarrantCancellation: [security_id:warrant_issuances]
  # etc.
```

**Pros:**

- Simpler implementation: only config changes + template updates
- No new data structures

**Cons:**

- Relies on naming convention to pick the right map
- Need to map `security_id` field to the correct issuance data field (it's `security_id` in the
  issuance data, not `id`)

### Option C: Computed Security ID Lookup

Keep current structure but add a helper function that searches all issuance maps:

```daml
findSecurityId : CapTableMaps -> Text -> Optional SecurityRef
findSecurityId maps secId =
  case Map.lookup secId (getAllSecurityIds maps.stock_issuances) of
    Some cid -> Some (StockSecurityRef cid)
    None -> case Map.lookup secId (getAllSecurityIds maps.warrant_issuances) of
      -- ... etc
```

**Pros:**

- No schema changes
- Works with existing structure

**Cons:**

- O(n) lookup across all maps (though typically small n)
- More complex validation logic in templates

## Recommended Approach

**Option A (Unified Securities Map)** provides the cleanest long-term solution with O(1) lookups and
type safety. The implementation complexity is manageable since:

1. The codegen already handles map updates for create/edit/delete
2. We just need to add `securities` map maintenance alongside existing map updates
3. The `SecurityRef` sum type provides clean pattern matching for type validation

- [ ] Add `SecurityRef` sum type to `Types.daml`
- [ ] Update `CapTable.daml.template` to include `securities: Map Text SecurityRef`
- [ ] Update `IssuerAuthorization.daml` to initialize empty `securities` map

### Phase 2: Update Issuance Processing

- [ ] Modify `create-case.daml` template for issuance types to:
  - Add entry to `securities` map when creating issuance
  - Validate `security_id` uniqueness across ALL issuance types (not just within type)
- [ ] Modify `edit-case.daml` template for issuance types to update `securities` map
- [ ] Modify `delete-case.daml` template for issuance types to remove from `securities` map

### Phase 3: Add Security ID Validation to Tier 4

- [ ] Update `captable-config.yaml` with new validation type for `security_id`
- [ ] Modify codegen to handle `security_id` validation using `securities` map
- [ ] Add validation to all tier 4 transactions:
  - Stock: Transfer, Cancellation, Acceptance, Retraction, Conversion, Repurchase, Reissuance,
    Consolidation
  - Warrant: Transfer, Cancellation, Acceptance, Retraction, Exercise
  - Convertible: Transfer, Cancellation, Acceptance, Retraction, Conversion
  - EquityCompensation: Transfer, Cancellation, Acceptance, Retraction, Exercise, Release, Repricing
  - Vesting: Start, Event, Acceleration

### Phase 4: Validation and Testing

- [ ] Add tests for security_id validation failures
- [ ] Add tests for cross-type security_id uniqueness
- [ ] Verify all existing tests pass
- [ ] Update documentation

## Success Criteria

- [ ] Creating an issuance with duplicate `security_id` (across any type) fails
- [ ] All existing tests pass
