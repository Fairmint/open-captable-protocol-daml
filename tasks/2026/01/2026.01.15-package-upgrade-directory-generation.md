# Task: Package Upgrade Directory Generation

**Date**: 2026-01-15 **Status**: Open **Target Repo**: `open-captable-protocol-daml`

## Goal

Eliminate verbose diffs from major version upgrades by generating versioned package directories at
build time instead of committing them directly to git.

## Problem

Currently, major version upgrades rename entire directories (e.g., `OpenCapTable-v27/` →
`OpenCapTable-v28/`). This causes:

- **Verbose diffs**: Every file shows as deleted and re-added
- **Hard to review**: Actual code changes are buried in path renames
- **Import churn**: All `daml.yaml` dependencies and imports reference the versioned folder name

Example of a typical major upgrade diff:

```
deleted:   OpenCapTable-v27/daml/Fairmint/OpenCapTable/CapTable.daml
deleted:   OpenCapTable-v27/daml/Fairmint/OpenCapTable/Types.daml
... (50+ files deleted)
added:     OpenCapTable-v28/daml/Fairmint/OpenCapTable/CapTable.daml
added:     OpenCapTable-v28/daml/Fairmint/OpenCapTable/Types.daml
... (50+ files added)
```

## Proposed Solution

Keep **stable source directory names** in git (e.g., `OpenCapTable/` without version suffix) and
generate the versioned directories at build time. The version lives only in `daml.yaml`.

### Directory Structure

**Before (current):**

```
/workspace/
  OpenCapTable-v28/      # Committed - version in folder name
    daml.yaml            # name: OpenCapTable-v28, version: 0.0.1
    daml/...
  multi-package.yaml     # references: OpenCapTable-v28
```

**After (proposed):**

```
/workspace/
  OpenCapTable/          # Committed - stable name, no version suffix
    daml.yaml            # name: OpenCapTable-v28, version: 0.0.1 (version HERE)
    daml/...

  generated/build/       # Gitignored - generated at build time
    OpenCapTable-v28/    # Copied + transformed from OpenCapTable/
      daml.yaml
      daml/...

  multi-package.yaml     # Auto-generated to generated/build/multi-package.yaml
```

### How It Works

1. **Source directories** have stable names (no version suffix in folder name)
2. **Version info** lives in each package's `daml.yaml` (package name + version)
3. **Build script** copies source → `generated/build/<package-name>/` (where `package-name` comes
   from `daml.yaml`)
4. **dpm builds** from `generated/build/` directory
5. **Upgrade script** only updates `daml.yaml` — no folder renames


**What changes:**

1. `OpenCapTable/daml.yaml` — name and version fields:

```yaml

```

**Investigation: What else references package names?**

Currently, 37 files reference `OpenCapTable-v28`. After migration, most references disappear
because:

| Reference Location                     | Current State      | After Migration               |
| -------------------------------------- | ------------------ | ----------------------------- |
| `multi-package.yaml`                   | `OpenCapTable-v28` | `OpenCapTable` (stable)       |
| `scripts/codegen/generate-captable.ts` | Hardcoded path     | Stable path                   |

**Key finding:** `scripts/packages.ts` currently hardcodes package names:

```typescript

```

**Fix required:** Modify `packages.ts` to read the `name` field from `daml.yaml` instead of
hardcoding it. After this fix, the only file that changes during a major upgrade is truly just
`daml.yaml`.

**Updated `packages.ts` approach:**

```typescript

  reports: { sourceDir: 'OpenCapTableReports' },
  // ...

function buildPackageConfig(def: { sourceDir: string }): PackageConfig {
  const damlYaml = readDamlYaml(def.sourceDir);
  return {
    name: damlYaml.name, // Read from daml.yaml, not hardcoded
    version: damlYaml.version,
    sourceDir: def.sourceDir,
    darName: damlYaml.name,
  };
}
```

**Diff is clean (after all fixes):**

```diff

- version: 0.0.3
+ version: 0.0.1
```

**Upgrade script simplification:** After migration, the upgrade script can be significantly
simplified — it only needs to update `daml.yaml` fields. No folder renames, no search/replace across
37 files.

## Technical Details

### `scripts/prepare-build.ts`

```typescript
/**
 * Prepares the generated/build directory for dpm compilation. Copies source packages to versioned
 * directories based on daml.yaml names.
 */

import * as fs from 'fs';
import * as path from 'path';
import * as yaml from 'yaml';

const ROOT = process.cwd();
const BUILD_DIR = path.join(ROOT, 'generated', 'build');
const SOURCE_MANIFEST = path.join(ROOT, 'multi-package.yaml');

interface DamlYaml {
  name: string;
  version: string;
}

function readDamlYaml(packageDir: string): DamlYaml {
  const yamlPath = path.join(packageDir, 'daml.yaml');
  return yaml.parse(fs.readFileSync(yamlPath, 'utf-8'));
}

function copyDir(src: string, dest: string): void {
  fs.mkdirSync(dest, { recursive: true });
  for (const entry of fs.readdirSync(src, { withFileTypes: true })) {
    const srcPath = path.join(src, entry.name);
    const destPath = path.join(dest, entry.name);
    if (entry.isDirectory()) {
      copyDir(srcPath, destPath);
    } else {
      fs.copyFileSync(srcPath, destPath);
    }
  }
}

function rewriteDependencies(buildPath: string, sourceToTargetMap: Map<string, string>): void {
  const yamlPath = path.join(buildPath, 'daml.yaml');
  const content = yaml.parse(fs.readFileSync(yamlPath, 'utf-8'));

  if (content.dependencies) {
    content.dependencies = content.dependencies.map((dep: string) => {
      if (dep.startsWith('../')) {
        const sourceDir = dep.replace('../', '');
        const targetDir = sourceToTargetMap.get(sourceDir);
        return targetDir ? `../${targetDir}` : dep;
      }
      return dep;
    });

    fs.writeFileSync(yamlPath, yaml.stringify(content));
  }
}

function main(): void {
  // Clean and create build directory with robust error handling
  try {
    fs.rmSync(BUILD_DIR, { recursive: true, force: true });
  } catch (error) {
    console.warn(`Warning: Could not clean build directory: ${error.message}`);
    // Continue - force: true should handle most cases, but log if something unusual happens
  }
  fs.mkdirSync(BUILD_DIR, { recursive: true });

  // Read source package list
  const manifest = yaml.parse(fs.readFileSync(SOURCE_MANIFEST, 'utf-8'));
  const buildPackages: string[] = [];
  const sourceToTargetMap = new Map<string, string>();

  // First pass: copy directories and build mapping
  for (const sourceDir of manifest.packages) {
    const sourcePath = path.join(ROOT, sourceDir);
    const damlYaml = readDamlYaml(sourcePath);

    // Target directory uses the name from daml.yaml (includes version)
    const targetDir = damlYaml.name;
    const targetPath = path.join(BUILD_DIR, targetDir);

    console.log(`${sourceDir}/ → generated/build/${targetDir}/`);
    copyDir(sourcePath, targetPath);
    buildPackages.push(targetDir);
    sourceToTargetMap.set(sourceDir, targetDir);
  }

  // Second pass: rewrite dependencies in copied daml.yaml files
  for (const sourceDir of manifest.packages) {
    const damlYaml = readDamlYaml(path.join(ROOT, sourceDir));
    const targetPath = path.join(BUILD_DIR, damlYaml.name);
    rewriteDependencies(targetPath, sourceToTargetMap);
  }

  // Generate build multi-package.yaml
  const buildManifest = { packages: buildPackages };
  fs.writeFileSync(path.join(BUILD_DIR, 'multi-package.yaml'), yaml.stringify(buildManifest));

  console.log(
    `\nGenerated generated/build/multi-package.yaml with ${buildPackages.length} packages`
  );
}

main();
```

## Migration Steps

The migration is designed to avoid breaking builds at any step:

1. **Create `prepare-build.ts` script** — Initially works with current versioned directory names
2. **Test build with current structure** — `multi-package.yaml` still uses versioned names like
   `OpenCapTable-v28`. The script reads `daml.yaml` from these directories and copies them to
   `generated/build/`. This validates the script works before any renames. (Note: `generated/` is
   already gitignored)
3. **Atomic rename commit** — In a single commit:
   - Rename `OpenCapTable-v28/` → `OpenCapTable/`
   - Update `multi-package.yaml` to use stable names: `OpenCapTable`
   - Update codegen scripts to use stable paths
4. **Update `upgrade-package.ts`** — Remove folder rename logic
5. **Test major upgrade produces clean diff**

> **Migration Note:** Step 4 must be atomic because the `prepare-build.ts` script reads directory
> names from `multi-package.yaml` and gets target names from each package's `daml.yaml`. After
> renaming directories, the source manifest must also be updated to reference the new stable names,
> but the `daml.yaml` files keep their versioned package names (which is the whole point).

## Edge Cases

### Codegen Paths

`generate-captable.ts` currently hardcodes `OpenCapTable-v28/`:

```typescript
const OPEN_CAPTABLE_DIR = path.join(REPO_ROOT, 'OpenCapTable-v28/daml/Fairmint/OpenCapTable');
```

Change to:

```typescript
const OPEN_CAPTABLE_DIR = path.join(REPO_ROOT, 'OpenCapTable/daml/Fairmint/OpenCapTable');
```

### Import Statements in DAML

DAML imports use module paths, not file paths:

```daml
import Fairmint.OpenCapTable.OCF.Stakeholder
```

These are **not affected** by directory renames — only the folder structure matters for `dpm`.

### Cross-Package Dependencies

Dependencies in `daml.yaml` use relative paths to sibling packages:

```yaml
# OpenCapTable/daml.yaml (source)
dependencies:
  - daml-prim
  - daml-stdlib
  - ../Shared
```

**Key insight:** The relative path `../Shared` works because:

1. `Shared/daml.yaml` has `name: Shared` (no version suffix)
2. `prepare-build.ts` copies `Shared/` → `generated/build/Shared/`
3. In `generated/build/`, the path `../Shared` from `OpenCapTable-v28/` still resolves correctly

For versioned packages referencing each other:

```yaml
# OpenCapTableReports/daml.yaml
dependencies:
  - ../OpenCapTable # Source path (stable)
```

After copying to `generated/build/`:

- `OpenCapTableReports-v01/` references `../OpenCapTable`
- But `generated/build/` has `OpenCapTable-v28/`, not `OpenCapTable/`

**Solution:** The `prepare-build.ts` script must rewrite dependency paths in copied `daml.yaml`
files, mapping source directory names to their build names. This is handled in the main script above
with a two-pass approach: first copy all directories and build the source→target mapping, then
rewrite dependencies in all copied `daml.yaml` files.

## Migration Safety

### Rollback Strategy

If the migration fails mid-implementation:

1. **Before directory renames (Phase 1-3):** Simply revert commits. Build continues to work with
   existing versioned directories.
2. **After directory renames (Phase 4+):** `git revert` the rename commit. The old versioned
   directories will be restored.

The key is that directory renames happen in a single atomic commit (step 4+5 together).

### CI/CD Impact

During transition:

- **No CI changes required for Phase 1-3** — existing builds work
- **Phase 4 commit must update CI if needed** — though `npm run build` abstracts the build process,
  so CI should "just work" if it uses npm scripts
- **Verify CI passes** before merging directory rename commit

### Local Development Workflow

After migration, developers should:

1. **Always run `npm run build`** — never run `dpm build` directly from source directories
2. **Edit files in source directories** (e.g., `OpenCapTable/`) — the `generated/build/` directory
   is ephemeral
3. **Ignore `generated/build/` directory** — it's gitignored and regenerated on each build

Add a note to `README.md` and `llms.txt` explaining the new workflow.
