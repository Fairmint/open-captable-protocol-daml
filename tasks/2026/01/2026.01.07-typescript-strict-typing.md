# Task: Eliminate `any` Types in TypeScript Scripts

**Date**: 2026-01-07
**Status**: Open
**Target Repo**: `open-captable-protocol-daml`

## Goal

Eliminate all uses of `any` and `unknown` types in the TypeScript codebase to align with strict typing guidelines. Per user rules: "Avoid `any` and `unknown`. Use precise, explicit types."

## Background

The repository has TypeScript scripts under `scripts/` that use `any` type assertions in multiple places. While these scripts work, they bypass TypeScript's type safety and make the codebase harder to maintain.

## Root Cause Analysis

### 1. SDK Type Mismatches (`utils.ts`)
The `@fairmint/canton-node-sdk` accepts string literals for `network` and `provider`, but the SDK types may be narrower than needed. Current code:
```typescript
network: network as any,  // 'mainnet' | 'devnet' | etc
provider: providerType as any,  // 'intellect' | '5n' | etc
```

**Solution Options**:
- A) Upstream fix: Contribute proper string union types to `canton-node-sdk`
- B) Local augmentation: Create `.d.ts` file extending SDK types
- C) Type guards: Create validated type predicates

### 2. JSON Parsing
Several files parse JSON without type validation:
```typescript
const data = JSON.parse(fs.readFileSync(path, 'utf8')) as any;
```

**Solution**: Use runtime validation (e.g., Zod) or define explicit interfaces

### 3. Error Handling
Catch blocks use `error: any`:
```typescript
} catch (error: any) {
```

**Solution**: Use `error: unknown` with type narrowing:
```typescript
} catch (error) {
  const message = error instanceof Error ? error.message : String(error);
}
```

### 4. Factory Data Objects
Factory scripts create objects typed as `any`:
```typescript
const reportsFactoryData: any = { ... }
```

**Solution**: Import or define proper interfaces from generated DAML types
