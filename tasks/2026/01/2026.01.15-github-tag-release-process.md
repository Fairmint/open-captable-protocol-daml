# Task: GitHub Tag-Based Release Process

**Date**: 2026-01-15 **Status**: Open **Target Repo**: `open-captable-protocol-daml`

## Goal

Implement an automated release process triggered by Git tags. Developers should be able to simply
push a version tag (e.g., `OpenCapTable-v28-v0.0.2`) and CI will automatically:

1. Build and test the package
2. Upload the DAR to the appropriate networks
3. Create a new factory contract if required (major version upgrade)
4. Update the NPM package

## Background

The current release process is manual and error-prone:

1. Run `npm run upgrade-package` for version bumps
2. Make code changes
3. Build and test locally
4. Run `upload-dar` for each network manually
5. Run `create-*-factory.ts` if major upgrade
6. Create PR and merge

This process has several pain points:

- **Easy to miss steps** — Forgetting to create a factory after major upgrade
- **Network-specific commands** — Must run upload for devnet AND mainnet separately
- **No audit trail** — Deployment history is scattered across release docs
- **Manual coordination** — Must remember which packages changed

## Proposed Solution

### Tag Format

Use full package names with semver tags for clarity and no configuration:

```
<package-name>-v<major>.<minor>.<patch>

Examples:
  OpenCapTable-v28-v0.0.2        # Minor update to OpenCapTable-v28
  OpenCapTable-v30-v0.0.1        # First release of new major version
  OpenCapTableReports-v01-v0.0.1 # OpenCapTableReports update
  CantonPayments-v0.1.0          # CantonPayments update
  CouponMinter-v0.0.2            # CouponMinter update
```

**Why full names?** Using the exact package directory name (e.g., `OpenCapTable-v28`) avoids
confusion and eliminates the need for alias mappings. The tag is self-documenting.

### Release Types

The CI determines release type by comparing the tag to the current state in `packages.ts` and
`daml.yaml`:

| Scenario          | Detection                                                                                                                                                    | Actions                         |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------- |
| **Major upgrade** | Resolved package directory for the tag's alias (via `packages.ts`) differs from the directory used by the last released tag for that alias (e.g., v28 → v29) | Upload DAR + Create new factory |
| **Minor/patch**   | Same package dir, version incremented                                                                                                                        | Upload DAR only                 |
| **No change**     | Tag version matches daml.yaml                                                                                                                                | Skip (already deployed)         |

In CI, "package directory changed" is computed as follows: (1) parse the tag to extract the package
name (e.g., `OpenCapTable-v28` from `OpenCapTable-v28-v0.0.2`), (2) verify this directory exists in
the repo, and (3) compare it to the directory from the most recent released tag for the same package
family. If the directory names differ (e.g., `v28` → `v29`), CI treats the tag as a **Major
upgrade**; if the directory is the same but the version has increased, it is a **Minor/patch**
release.

### Factory Creation Rules

A new factory is needed when:

1. **Major package upgrade** — Package directory name changes (contains version suffix)
2. **First deployment** — No factory exists for this package version

Factory is NOT needed when:

1. **Minor/patch version** — Same package, different internal version
2. **Re-deployment** — Same version already deployed to this network

### CI Workflow

New workflow: `.github/workflows/release.yml`

```yaml
name: Release

on:
  push:
    tags:
      - 'OpenCapTable-v*'
      - 'OpenCapTableReports-v*'
      - 'CantonPayments-v*'
      - 'CouponMinter-v*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          lfs: true

      - name: Parse release tag
        id: parse-tag
        run: |
          # Extract package and version from tag
          TAG="${GITHUB_REF#refs/tags/}"
          # ... parsing logic

      - name: Install DAML SDK
        run: |
          # TODO: Install DAML SDK for this repository.
          # See .github/workflows/publish.yml for the full installation script.
          echo "Installing DAML SDK (see .github/workflows/publish.yml for full script)"

      - name: Build and test
        run: |
          npm install
          npm run build
          npm run test

      - name: Deploy to devnet
        run: |
          npm run upload-dar -- --package ${{ steps.parse-tag.outputs.package }} --network devnet

      - name: Deploy to mainnet
        run: |
          npm run upload-dar -- --package ${{ steps.parse-tag.outputs.package }} --network mainnet

      - name: Create factory (if major upgrade)
        if: steps.parse-tag.outputs.needs_factory == 'true'
        run: |
          npx tsx scripts/create-${{ steps.parse-tag.outputs.factory_script }}.ts --network devnet
          npx tsx scripts/create-${{ steps.parse-tag.outputs.factory_script }}.ts --network mainnet

      - name: Update NPM package
        run: npm publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

## Release Workflow Example

### Minor Update (No Factory)

```bash
# 1. Make changes to DAML code
# 2. Update version in daml.yaml: 0.0.1 → 0.0.2
# 3. Commit and push
git add .
git commit -m "feat: add new validation to StockIssuance"
git push

# 4. Create and push tag (use full package name)
git tag OpenCapTable-v28-v0.0.2
git push origin OpenCapTable-v28-v0.0.2

# CI automatically:
# - Builds all packages
# - Runs tests
# - Uploads DAR to devnet and mainnet
# - Publishes NPM package
```

### Major Upgrade (With Factory)

```bash
# 1. Run upgrade script
npm run upgrade-package -- --package OpenCapTable --type major
# Creates OpenCapTable-v30/ from OpenCapTable-v28/

# 2. Make breaking changes
# 3. Commit and push
git add .
git commit -m "feat!: breaking change to EditIssuer return type"
git push

# 4. Create and push tag (new package directory = new factory)
git tag OpenCapTable-v30-v0.0.1
git push origin OpenCapTable-v30-v0.0.1

# CI automatically:
# - Builds all packages
# - Runs tests
# - Uploads DAR to devnet and mainnet
# - Creates new OcpFactory contract on both networks
# - Saves factory contract IDs to generated/
# - Commits updated generated files
# - Publishes NPM package
```

## Edge Cases

### 1. Tag Without daml.yaml Update

If tag version doesn't match `daml.yaml`, CI should fail early with clear error:

```
❌ Tag version (0.0.3) doesn't match daml.yaml version (0.0.2)
   Update daml.yaml first, then create the tag.
```

### 2. Re-tagging Same Version

If DAR is already uploaded (as recorded in the corresponding `metadata.json` files in `dars/` via
`recordNetworkUpload()`), skip upload but still update NPM:

```
⏭️  DAR already uploaded to devnet, skipping
⏭️  DAR already uploaded to mainnet, skipping
📦 Publishing NPM package...
```

### 3. Factory Already Exists

Query the ledger or check `generated/` files before creating factory:

```
⏭️  Factory already exists for OpenCapTable-v30, skipping creation
```

### 4. Partial Failure

If devnet succeeds but mainnet fails:

- Don't create factory (needs both networks)
- CI marks as failed
- Manual intervention required

**Note on `publish.yml` and tag namespaces**: The existing `.github/workflows/publish.yml` workflow
currently creates generic `v*` tags after NPM publishes (for example, via
`git tag -a "v$NEW_VERSION"` on line 78). The new tag-based release workflow proposed in this
document will instead be triggered by package-scoped tags like `OpenCapTable-v28-v0.0.2`. As part of
this task, confirm that:

- `publish.yml` continues to own the generic `v*` tags used for NPM publishing, and
- the new release workflow is configured to trigger only on the package-scoped tags
  (`OpenCapTable-v*`, `CantonPayments-v*`, etc.), so that the two workflows do not conflict.

These namespaces are distinct and won't conflict:

- `v*` tags → NPM version tracking (created by `publish.yml`)
- `OpenCapTable-v*`, `CantonPayments-v*`, etc. → DAR deployments (triggers `release.yml`)

## Security Considerations

1. **Secrets management** — Auth tokens for devnet/mainnet stored as GitHub secrets
2. **Tag protection** — Consider branch protection rules for tags
3. **Audit trail** — All deployments tracked via Git tags and GitHub Actions logs
4. **Rollback** — Previous tags remain valid; can re-deploy old versions

## Open Questions

1. **Network selection**: Should tags deploy to both networks or allow selection?
   - `OpenCapTable-v28-v0.0.2` → both networks (default)
   - `OpenCapTable-v28-v0.0.2-devnet` → devnet only (for testing)
   - **Recommendation**: Default to both, suffix for single network

2. **Approval gates**: Should mainnet require manual approval?
   - GitHub Environments can require approval before deployment
   - Adds friction but increases safety
   - **Recommendation**: Start without, add if needed
