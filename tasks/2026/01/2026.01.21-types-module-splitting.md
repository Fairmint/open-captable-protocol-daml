# Task: Split Types.daml into Domain-Focused Modules

**Date**: 2026-01-21 **Status**: Open **Target Repo**: `open-captable-protocol-daml`

## Goal

Split the large `Types.daml` file (1000+ lines) into smaller, domain-focused modules for better
maintainability, discoverability, and compile times.

## Problem

`OpenCapTable-v29/daml/Fairmint/OpenCapTable/Types.daml` has grown to over 1000 lines containing:

- Core types (`Context`, `TransferOperation`)
- Monetary types (`OcfMonetary`, `OcfTaxID`)
- Conversion mechanism types (SAFE, Note, Ratio, etc.)
- Stakeholder types
- Vesting types
- Validation helpers

This causes:

- **Hard to find types**: Developers must search through a large file
- **Import bloat**: Files import everything even when they need one type
- **Compile time**: Changes trigger recompilation of all dependents
- **Cognitive load**: Understanding relationships between types is difficult

## Solution

Delete `Types.daml` and split into domain-focused modules under a `Types/` subdirectory. Update all
imports across the codebase to use the new specific modules directly.

### New Module Structure

```
OpenCapTable-v29/daml/Fairmint/OpenCapTable/
└── Types/
    ├── Core.daml           # Context, TransferOperation, basic validators
    ├── Monetary.daml       # OcfMonetary, OcfTaxID, OcfAddress
    ├── Contact.daml        # OcfEmail, OcfPhone, OcfPhoneType
    ├── Conversion.daml     # All conversion mechanism types
    ├── Stock.daml          # Stock-related types and enums
    ├── Stakeholder.daml    # Stakeholder relationship types
    └── Validation.daml     # Shared validation helpers
```

**Note**: No re-export module. Each file imports exactly what it needs from the specific modules.

## Risk Assessment

**Medium risk** — Requires updating imports across many files:

- Many files need import changes
- Must update all files atomically to avoid build failures
- Potential for missed imports

**Mitigation:**

- Use grep/search to find all imports of `Types`
- Make all changes in one commit
- Run build frequently during development
- Easy to verify: build fails immediately if any import is wrong
