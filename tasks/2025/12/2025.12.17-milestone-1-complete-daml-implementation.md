## Implementation Pattern

For each new type (or batch of related types):

1. **Query production database** for real examples (see
   [Test Data from Production](#test-data-from-production))
2. Create DAML module in `OpenCapTable-v30/daml/Fairmint/OpenCapTable/`
   - **Add module header** with OCF schema link (see `llms.txt` for format)
3. Define data type matching OCF JSON schema (field order matters)
4. Add choice to `Issuer.daml` for creation (alphabetical order)
5. Add validators in `Types.daml` if new enums/types needed
6. Create test in `Test/` directory (use `TypeName_Action` naming pattern)
   - **Use realistic test data** based on production examples (anonymized)
7. Update `OCF_IMPLEMENTATION_STATUS.md`
8. Update this task file's Changelog (**all PRs must be logged**)
9. Extract durable patterns to
   [ADR-001](https://github.com/Fairmint/canton/blob/main/docs/developer/adr/001-ocf-captable-on-canton.md)
   when completing a category

## Test Data from Production

Before implementing tests, query the databases for real examples:

```sql
-- Check what examples exist for a transaction type
SELECT ocf_data
FROM ocf_objects
WHERE subtype = 'TX_<TYPE_NAME>'
LIMIT 5;

-- Or search by pattern
SELECT subtype, COUNT(*)
FROM ocf_objects
WHERE subtype LIKE 'TX_%<PATTERN>%'
GROUP BY subtype;
```

**Database Priority:**

1. **Check prod first** (MCP `postgres-prod`) - preferred for realistic data
2. **Fall back to dev** (MCP `postgres-dev`) - if prod has no examples
3. **Extrapolate from similar types** - if neither has examples

**Guidelines:**

- Query the `ocf_objects` table; `ocf_data` column contains full OCF JSON
- **⚠️ REQUIRED: Anonymize all PII** before including in documentation:
  - Replace real names with placeholders (e.g., "Founder A", "Employee B", "Investor A")
  - Replace company names with generic names (e.g., "ExampleCo Inc.", "ABC Holdings Ltd")
  - Keep UUIDs and other identifiers as-is (not considered PII)
- Keep realistic values: amounts, quantities, dates, field patterns
- When extrapolating, use the same security type's issuance/cancellation as a pattern

## Implementation Insights

Lessons learned during implementation to help with future types:

### TX_CONVERTIBLE_CANCELLATION (2025-12-23)

**Key difference from TX_STOCK_CANCELLATION:**

- Uses `amount: OcfMonetary` (monetary value) instead of `quantity: Decimal` (share count)
- Convertibles represent monetary instruments, not shares

**Test patterns:**

- Create prerequisite issuance before cancellation (follows real workflow)
- Use realistic data from production database for test values
- Test both `ArchiveByIssuer` (single signer) and `Archive` (both signatories)
- Include test case with optional fields populated (partial cancellation scenario)

**Documentation:**

- Always include OCF schema URL in comments above data type
- Update summary table counts in `OCF_IMPLEMENTATION_STATUS.md`
- Add changelog entry with date and new totals

### TX_WARRANT_CANCELLATION and TX_EQUITY_COMPENSATION_CANCELLATION (2025-12-23)

**Pattern identical to TX_STOCK_CANCELLATION:**

- Both use `quantity: Decimal` (unit count) like stock cancellations
- Structurally identical - can be implemented in parallel

**Batch implementation approach:**

- When multiple types share the same schema structure, implement together
- Reduces context switching and ensures consistency
- Cancellation category now 100% complete (4/4 types)

**Production data usage:**

- Used real production TX_EQUITY_COMPENSATION_CANCELLATION data for realistic tests
- Example: 214,200 options cancelled after exercises, with contextual comments

### Transfer Transactions (2025-12-24)

**Pattern identical to Cancellations with key differences:**

- Transfers have `resulting_security_ids: [Text]` (required, min 1 item) - IDs of new securities
  created
- Transfers have `consideration_text: Optional Text` instead of `reason_text: Text`
- Stock/Warrant/EquityCompensation use `quantity: Decimal`
- Convertible uses `amount: OcfMonetary` (consistent with other convertible transactions)

**Production data analysis:**

- Checked both prod and dev databases for examples
- Found 2 TX_STOCK_TRANSFER and 1 TX_CONVERTIBLE_TRANSFER in prod
- No TX_WARRANT_TRANSFER or TX_EQUITY_COMPENSATION_TRANSFER (rare in practice)
- Used realistic data patterns for tests, anonymizing PII

**Key insight for transfers:**

- Warrant transfers are rare (warrants usually exercised, not transferred)
- Equity compensation transfers are very rare (options typically non-transferable except NSOs to
  family trusts)
